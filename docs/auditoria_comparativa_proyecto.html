2Decisión arquitectónica

Arquitectura obligatoria — NO negociable:

FastAPI backend + API REST + frontend HTML/JS/Plotly

  • NO migrar el frontend a Streamlit, Dash, React, Vue, Svelte ni ningún otro framework.
  • NO duplicar lógica financiera pesada en el frontend.
  • El HTML solo: consume endpoints, grafica con Plotly, muestra tablas, interpreta resultados.
  • El backend hace: cálculos, validaciones, persistencia, ML, todo lo pesado.

Responsabilidades por capa

CapaResponsabilidadTecnologías

Reglas de oro de la arquitectura

  1. Toda lógica financiera vive en api/services/ o api/logic.py — nunca en JS.
  2. Toda entrada del usuario pasa por Pydantic antes de llegar al cálculo.
  3. Toda persistencia pasa por SQLAlchemy ORM con sesión inyectada (Depends(get_db)).
  4. Todo endpoint nuevo aparece en /docs con response model tipado.
  5. Toda llamada a API externa (Yahoo, FRED) pasa por servicio con cache en BD.
  6. Endpoints actuales no se borran — se agregan alias cortos si el nuevo instructivo lo prefiere.

Objetivo

Construir las cuatro capas financieras nuevas exigidas por el instructivo: Renta Fija (curva FRED + Nelson-Siegel + duración + convexidad), Opciones (Black-Scholes + 5 Greeks + paridad), Stress Testing (3 escenarios + bar chart + heatmap), y ML Pipeline (train + Singleton + /predict + logging en PredictionLog).

Qué se debe hacer

M9 Renta Fija

  • Crear api/services/yield_curve.py con clase YieldCurve: fit_nelson_siegel(maturities, yields), spot_rate(tau)
  • Crear api/services/bond.py con clase Bond: macaulay_duration(), modified_duration(), convexity()

7Riesgos técnicos y cómo evitarlos

RiesgoImpactoMitigación
Migrar mal a SQLAlchemy y romper el login Alto Cohabitación de capas durante la transición. Eliminar sqlite3 directo solo al final.
FRED API key falta o cuota agotada Medio Cache obligatorio con TTL 24h. Fallback a yfinance si FRED falla.
Cambiar URLs y romper el dashboard Alto NO cambiar URLs existentes. Solo agregar alias cortos si es necesario.
Dashboard se rompe al cargar nuevos cómputos pesados Medio Mantener data.js funcional siempre. Nuevos módulos cargan vía API en demanda.
data.js crece > 3 MB con nuevos módulos Medio No incluir Renta Fija y Opciones en snapshot estático — solo via API en vivo.
Imagen Docker grande con sklearn Bajo Multi-stage en Fase 5. Usar slim-bookworm.
Cold start en Render durante sustentación Medio Hacer warmup 1 minuto antes de la demo. Documentar en README.
Tests dependen de datos externos (yfinance / FRED) Medio Tests usan BD en memoria + datos sintéticos, no APIs externas.
Singleton mal implementado (recarga por request) Alto Verificar log "modelo cargado" aparece UNA sola vez al arrancar uvicorn.
JWT_SECRET o FRED_API_KEY filtrado al repo Alto Solo .env.example al repo. Nunca el .env real. Configurar en Render via UI.

  3. Funcionalidad agregada: bullet list de qué quedó funcionando
