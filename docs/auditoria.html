5Brecha entre proyecto actual y nuevo instructivo

Lo que falta agregar desde cero

  • Capa SQLAlchemy completa: migrar api/database.py de sqlite3 a SQLAlchemy ORM. Crear modelos Asset, Price, Portfolio, PredictionLog. Implementar get_db() generator y Depends(get_db) en endpoints.
  • Módulo Renta Fija (M9): clase YieldCurve con fit_nelson_siegel(), clase Bond con duración Macaulay/modificada y convexidad. Endpoints /curva-rendimiento y /bono/duracion. Integración con FRED (DGS3MO, DGS1, DGS2, DGS5, DGS10, DGS30).
  • Módulo Opciones (M10): clase OptionPricer con black_scholes(), greeks(), implied_volatility() (Newton-Raphson). Endpoint /opcion/precio con response anidado.
  • Módulo Stress (M11): clase StressTester con escenarios tipados (rate, market, volatility, combinado). Endpoint /stress que recibe portafolio + lista de escenarios.
  • Componente ML: script train.py, clase Singleton ModelPredictor, endpoint /predict con logging en PredictionLog. Justificación del problema analítico.
  • Suite pytest: tests/conftest.py con BD en memoria + override de get_db. 5 tests mínimos.
  • GitHub Actions CI: .github/workflows/ci.yml con pytest.
  • FRED API integration: nueva clase de servicio para consumir FRED con cache en SQLite (FRED_API_KEY en .env).

Lo que toca reforzar

  • Dockerfile multi-stage para reducir tamaño de imagen.
  • EWMA en M3 con λ=0.94 default y configurable vía query.
  • QP explícito en M6 con cvxpy o scipy.optimize.minimize (no solo simulación).
  • Comparación con/sin no-negatividad en frontera eficiente.
  • Persistencia de señales del M7 en tabla signals_log.
  • Decorador propio para latencia en endpoints críticos (/predict).
  • Deploy real verificado en Render con URL pública en README.

Lo que toca rehacer

  • Capa de persistencia — migración completa de sqlite3 directo a SQLAlchemy ORM (cambio estructural, requiere ajustar todos los queries).
  • Esquema de URLs — el nuevo instructivo prefiere rutas cortas (/precios) en lugar de las versionadas largas (/api/v1/technical). Decisión recomendada: Opcional — mantener las actuales y agregar alias cortos, no romper compatibilidad.

Lo que sobra o quedó desalineado

  • El endpoint /api/v1/all que devuelve todo de una vez no aparece en el nuevo instructivo. Útil para el frontend, pero ahora sería redundante con endpoints separados. Mantener por compatibilidad.
  • El panel macro actual usa ^IRX, ^TNX, ^GSPC de yfinance. El nuevo instructivo exige FRED + opcionalmente Banco de la República. Toca migrar la fuente, no solo agregar.

9Riesgos y puntos delicados

Crítico Cosas que pueden romper el proyecto si se tocan mal

  • Migración a SQLAlchemy: el código actual de api/database.py es funcional. Migrar mal puede romper el login y la creación de usuarios. Estrategia: crear los modelos ORM en paralelo, hacer cohabitar las dos capas durante la migración, no eliminar el código viejo hasta validar el nuevo.
  • Cambio de fuente macro (yfinance → FRED): requiere obtener API key gratuita de FRED, agregar variable de entorno, y manejar el caso de cuota agotada. Mantener fallback a yfinance.
  • Esquema de URLs nuevo: si se cambian todos los endpoints (/api/v1/risk → /var), se rompe el dashboard. Recomendación: NO cambiar esquema, agregar alias.
  • Limpieza de dashboard: agregar 3 tabs nuevas sin romper el flujo de carga estática. El data.js debe seguir funcionando incluso sin nuevos módulos.
  • generate_data.py: si se agregan nuevos cómputos (Renta Fija, Opciones), el snapshot crece. Cuidar el tamaño del data.js (actualmente 1.7 MB, no superar 3 MB).

Mantener Decisiones actuales que conviene NO cambiar

  • Frontend HTML + Plotly puro (sin React/Vue)
  • Arquitectura híbrida estático + API
  • JWT + BCrypt para autenticación
  • Pydantic v2 con validators y model_validators
  • Inyección de dependencias con Annotated[T, Depends(...)]
  • Selección de portafolio NU·AMZN·SONY·XOM·WPM (a menos que el equipo decida explícitamente cambiarlo)
  • Benchmark ^GSPC con justificación documentada
  • Estructura de carpetas ya limpia (api/, dashboard/, docs/, tests/, data/)

Otros riesgos identificados

  • Tamaño de imagen Docker con sklearn: agregar +30 MB. Manejable con multi-stage.
  • Cold start en Render free tier: ~30s tras 15 min sin tráfico. Documentar y hacer warmup antes de demo.
  • FRED API rate limit: 120 req/min sin key. Cache obligatorio en SQLite con TTL 24h.
  • Singleton ML: si el modelo crece (TF/Keras), la imagen Docker pasa de 150 MB a 500+ MB. Considerar tensorflow-cpu en lugar de tensorflow.

Frente al nuevo instructivo, el avance es significativo pero no completo (~55%). La diferencia no es de pulido — es de ampliación funcional sustancial.

Distancia al nuevo instructivo

El proyecto está a ~13-16 días de trabajo enfocado de cumplir el nuevo instructivo. No es un retrabajo masivo, sino una expansión disciplinada: se añaden 4 módulos financieros + capa ORM + tests + ML + DevOps mejorado, sin tocar lo que ya funciona.

¿Es ampliación incremental o expansión fuerte?

Expansión fuerte. El instructivo nuevo cambia el carácter del proyecto: pasa de ser una plataforma de análisis de riesgo de mercado a ser una plataforma integral con renta fija, derivados, stress testing y ML servido en producción. La arquitectura debe sostener esa expansión, lo que justifica la migración a SQLAlchemy y el endurecimiento de tests/CI/Docker.

Mejor estrategia para completar sin dañar lo construido

  1. Trabajar en una rama separada (feature/instructivo-iii) y mergear solo al validar cada fase.
  2. Respetar el orden de fases: refuerzo → ORM → módulos nuevos → ML/tests/deploy → docs. No saltar fases.
  3. Migrar SQLAlchemy en paralelo, no en reemplazo: dejar que la capa nueva conviva con la vieja durante la transición.
  4. Cada módulo nuevo se prueba aislado en /docs antes de integrarlo al dashboard.
  5. Mantener data.js funcional siempre: el dashboard nunca debe quedar dependiente de la API durante el desarrollo.
  6. Documentar a medida que se desarrolla: actualizar README y CLAUDE.md tras cada fase, no al final.

Lo que el equipo ya tiene es defendible hoy. Lo que falta es expansión, no corrección. El plan de fases respeta el principio fundamental: nunca romper lo que funciona.
